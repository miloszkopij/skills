# Intellectual Property & Licensing Reference

For an IT and retail buyer, IP issues come up in three places: ownership of work product, open-source license compliance, and IP indemnification.

---

## 1. Ownership of Inputs and Outputs (SaaS / AI / Dev Services)

The clean buyer-favourable structure:
- **Inputs** (buyer's data, prompts, source materials) → buyer retains all rights, vendor gets only a limited license to process.
- **Outputs** (results generated by service) → buyer owns, or vendor assigns to buyer.
- **Vendor's underlying tech** (the service itself, models, software) → vendor keeps; buyer gets a use license during the term.

| Found | Severity |
|---|---|
| Buyer retains Inputs, owns Outputs, vendor disclaims rights to both | 🟢 ACCEPTABLE |
| Buyer retains Inputs, Outputs ownership unclear or shared | 🟠 NEGOTIATE — clarify |
| Vendor takes a license to Inputs to train models | 🔴 DEAL_BREAKER for IT buyers (data exfiltration risk); 🟠 if narrowly scoped to improving service to buyer only and revocable |
| Vendor retains ownership of Outputs / requires assignment back to vendor | 🔴 DEAL_BREAKER |
| Vendor may use Inputs for "any business purpose" | 🔴 DEAL_BREAKER |

### Polish law on copyright assignment

**Critical:** Under Polish Copyright Act (Ustawa o prawie autorskim i prawach pokrewnych), Art. 41 and Art. 53:
- Copyright assignment requires **written form under pain of nullity** (forma pisemna pod rygorem nieważności).
- The contract must specify **fields of exploitation** (pola eksploatacji) — explicitly listed, not vague.
- Future works can only be assigned in limited circumstances (Art. 41 §3 — within the scope of a specifically defined work).

Implications:
- If a contract assigns IP to/from a Polish party but is only signed electronically without qualified e-signature, the IP assignment may be void (the rest of the contract usually survives).
- If "fields of exploitation" are not enumerated, the assignment is limited only to fields directly required for the contract's purpose — and that's narrowly construed.
- Flag electronic-only signature for any contract containing IP assignment to/from a Polish entity as 🟠 NEGOTIATE — recommend qualified e-signature or wet signature on the IP assignment portion.

### Moral rights (autorskie prawa osobiste)

Moral rights in Polish law are **inalienable** — cannot be assigned or waived. The most a contract can do is the author agreeing not to exercise them, or authorising a third party to do so. Flag any clause that purports to assign or waive moral rights wholesale as ⓘ INFORMATIONAL (it's not effective, but not fatal).

## 2. Open-Source Licenses

When the contract mentions OSS components, dependencies, or covers software delivered to the buyer, check OSS license compatibility against the buyer's intended use.

### License taxonomy

| License | Type | Buyer impact | Severity if found in delivered software |
|---|---|---|---|
| **MIT** | Permissive | Include notice. Use freely. | 🟢 ACCEPTABLE |
| **BSD-2-Clause, BSD-3-Clause** | Permissive | Same as MIT, plus no-endorsement clause for BSD-3. | 🟢 ACCEPTABLE |
| **Apache-2.0** | Permissive | Include notice + NOTICE file. Patent grant included. State changes. | 🟢 ACCEPTABLE |
| **ISC** | Permissive | Like MIT. | 🟢 ACCEPTABLE |
| **Unlicense, 0BSD, CC0** | Public-domain-like | No restrictions. | 🟢 ACCEPTABLE |
| **LGPL-2.1, LGPL-3.0** | Weak copyleft | Can link dynamically without infecting proprietary code. Static linking triggers copyleft. | 🟢 ACCEPTABLE for dynamic; ⓘ for static |
| **MPL-2.0** (Mozilla) | Weak copyleft (file-level) | Modifications to MPL files must be released. Combining with proprietary OK. | 🟢 ACCEPTABLE |
| **EPL-2.0** | Weak copyleft | Similar to MPL. | 🟢 ACCEPTABLE |
| **GPL-2.0, GPL-3.0** | Strong copyleft | Distributing software linking to GPL code requires releasing your code under GPL. | 🟠 NEGOTIATE — must isolate from proprietary code; 🔴 if delivered as part of a proprietary product |
| **AGPL-3.0** | Network copyleft | Even SaaS use triggers copyleft — must release source to users of the network service. | 🔴 DEAL_BREAKER for proprietary SaaS; 🟠 if internal-only use |
| **SSPL** (MongoDB), **BUSL** (Business Source) | Source-available, not OSI | Commercial use often restricted. | 🟠 NEGOTIATE — read the specific license; check the "Change Date" for BUSL |
| **Elastic License v2, Confluent Community License** | Source-available | Restrictions on competing with the vendor. | 🟠 NEGOTIATE if buyer might compete |
| **Commons Clause** | Add-on restriction | Cannot sell the software as a product. | 🟠 NEGOTIATE |
| **WTFPL, Beerware** | Permissive but legally weak | Compliance risk for risk-averse buyers. | ⓘ INFORMATIONAL |
| **Custom / proprietary "open source"** | Read carefully | Often hides restrictions. | 🟠 NEGOTIATE — read every word |

### Common OSS-related issues to flag

- **Contract states "software is free of third-party rights"** but the software ships with GPL components → vendor's warranty is false. 🟠 NEGOTIATE — get explicit OSS notice schedule.
- **Buyer must indemnify vendor for "any IP claims"** without OSS carve-out → unfair given buyer can't audit OSS provenance. 🟠 NEGOTIATE.
- **No SBOM (Software Bill of Materials) provided** → for IT buyer subject to NIS2 / DORA / CRA, this may become a compliance issue. 🟠 NEGOTIATE — request SBOM clause.
- **No security patch SLA for OSS components** → flag if delivered software has OSS deps. 🟠 NEGOTIATE.
- **EU Cyber Resilience Act (CRA)** — applies from late 2027 to products with digital elements sold in EU. Vendor's contract should anticipate this; flag if there's no provision for compliance updates. ⓘ INFORMATIONAL for now.

## 3. Trademark

If buyer's trademark is licensed to vendor (e.g., for vendor to display "powered by buyer" or to brand the service):
- Scope, territory, duration must be specified.
- Quality control clause is necessary (otherwise trademark may be at risk of being abandoned under PL/EU TM law).
- Termination must include cessation of all trademark use.

For typical SaaS where vendor only displays buyer's logo on their customer page → publicity clause covers it (see EU/Poland checklist §10).

---

## 4. Patent

Most B2B IT contracts don't deeply involve patents. Watch for:
- **Patent indemnity** — should be included in vendor's IP indemnification (see EU/Poland §5).
- **Mutual covenant not to sue** for patent infringement — common in standards bodies; rare in commercial SaaS.
- **Patent assignment of work product** — for dev services, ensure patent rights (along with copyright) are part of the assignment.
- **Apache-2.0** explicitly grants a patent license to recipients (and has a defensive termination clause if the recipient sues the contributor for patent infringement). 🟢 — good for buyers.

## 5. AI-specific IP issues (relevant for IT buyer)

For contracts involving AI / LLM services:

- **Training-data IP indemnity** — vendor should indemnify against claims that its training data infringed third-party IP and that outputs derived from it infringe. (Major vendors now do — e.g., Anthropic, OpenAI, Microsoft, Google — but check.) Missing = 🟠 NEGOTIATE.
- **Output IP ownership** — buyer should own outputs (subject to inherent limitations: outputs may not be copyrightable if not creative enough, may not be patentable if not novel, etc.). Flag if vendor claims rights to outputs.
- **No-train commitment** — vendor must commit not to train on buyer's data (or specifically opt-in for it). Missing = 🟠 (sometimes 🔴 depending on data sensitivity).
- **Hallucination disclaimer** — vendor will disclaim accuracy of outputs. Standard. 🟢 ACCEPTABLE but flag for ⓘ awareness.
- **Trademark / right-of-publicity claims arising from outputs** — usually carved out of indemnity. 🟠 NEGOTIATE if buyer plans to use outputs in marketing.

## 6. Specific things to look for in licensing language

Red phrases that warrant a closer look:

- "perpetual, irrevocable, worldwide, royalty-free, sublicensable license to [buyer's] content" → potentially overbroad; check scope.
- "transferable license" (vendor wants right to transfer license to acquirer) → usually OK but flag for ⓘ.
- "derivative works" rights → check that they go in the buyer's direction (for service outputs) not against the buyer (over the buyer's content).
- "for any business purpose" → too broad for a license to buyer's content. 🔴.
- "to improve our service" → ambiguous; could mean training, telemetry, anything. 🟠 NEGOTIATE — narrow to specifics.
